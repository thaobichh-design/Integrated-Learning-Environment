| **EPS.P1: Agents are specialists, not generalists**     | This principle governs ALL agent design. An agent with one clear job, 2-3 tools, and precise instructions will outperform a generalist agent with 20 tools and vague instructions. This is the most violated principle by beginners.    | Each agent has ONE role (e.g., 'Senior Financial Analyst'), a bounded tool set (e.g., YFinance fundamentals only), and a structured output schema (e.g., {revenue_growth, margin, debt_ratio, valuation, confidence}). The agent doesn't decide WHAT to analyse — the Coordinator does. The agent executes its speciality. | Works when: agents are defined with job-description-level specificity — role, tools, output schema, constraints, failure behaviour. The Coordinator knows each agent's capability and assigns work accordingly.                     | Rooted in: division of labour (Adam Smith), microservices architecture (single responsibility principle), cognitive load theory (focused context → better reasoning).                         | Specialist agents have focused context → LLM reasoning is higher quality per task. Focused tool sets → fewer tool selection errors. Structured output → reliable handoffs. All three effects compound in multi-agent teams.                                                                                   | Division of labour, single responsibility principle, cognitive load theory, Unix philosophy ('do one thing well'). | Agno Agent class (forces model + tools + instructions definition), output schema validation (Pydantic models), Coordinator Mode (assigns work by capability).                                   | Culture of 'good enough specialisation' — the temptation is always to add more tools to one agent instead of creating a new specialist. The environment must reward small, focused agents. | Violated when: User creates one 'super agent' with 15 tools and instructions like 'help me with investment research'. The agent hallucinates tool selection, produces unstructured output, and can't be debugged. | Temptation of simplicity — one agent is simpler to CREATE but harder to OPERATE. Multiple specialists are harder to create but easier to debug, improve, and compose.                                            | One super-agent fails non-deterministically (sometimes picks wrong tool, sometimes right). Debugging is impossible because the failure mode changes each run. The User can't isolate which capability is broken.                                                                | 'More tools = more capable' (more tools = more confusion for the LLM). 'One agent is simpler' (simpler to create, not simpler to operate).                                                                                             | A single agent with unbounded tool access and vague instructions. No output schema validation.                                                 | Rushing to 'get it working' without designing agent roles. No time allocated for role decomposition before coding.                        | Split the super-agent. Identify its 3 main capabilities. Create 3 specialist agents. Wire them through a Coordinator. Compare output quality.                                                                                          | Before building ANY agent, write a 1-paragraph job description: role, tools, output format, constraints. If it takes more than 1 paragraph, split into 2 agents.                         |
| **EPS.P2: The orchestrator directs; it never does**     | The Coordinator/Director agent (or the User) decides WHAT to do and WHO does it. It never calls tools directly or produces domain-specific analysis. Mixing orchestration and execution creates a single point of failure.              | Separation of direction from execution. The Coordinator reads the goal, decomposes it into sub-tasks, assigns each to the most capable specialist, tracks progress, handles failures, and assembles the final output. It is the 'manager' — not an 'individual contributor'.                                               | Works when: the Coordinator's only tools are 'delegate to Agent X' and 'read Agent X's output'. It has no domain tools (no YFinance, no web search). Its intelligence is in task decomposition and delegation.                      | Rooted in: management theory (span of control), separation of concerns (architecture pattern), orchestration vs choreography (distributed systems).                                           | Clean separation means: Coordinator failures are ALWAYS about task decomposition (fixable by improving the Coordinator's instructions). Specialist failures are ALWAYS about domain execution (fixable by improving the specialist's tools/instructions). No ambiguity about WHERE a failure occurred.        | Separation of concerns, span of control, orchestration pattern, command pattern.                                   | Agno Coordinator Mode (built-in delegation pattern), Team class with mode='coordinator', structured inter-agent messaging.                                                                      | Discipline to NOT add domain tools to the Coordinator. Clear role boundaries in the team definition. Testing that verifies the Coordinator never calls domain tools directly.              | Violated when: the Coordinator also has domain tools and 'helps out' when specialists are slow. This makes it a generalist (violates EPS.P1) and a bottleneck.                                                    | Efficiency temptation — 'the Coordinator already has the context, why not just let it do the analysis too?' This collapses the separation and makes debugging impossible.                                        | When the Coordinator does domain work, failures become ambiguous: was it a decomposition error or an analysis error? The User can't tell. Debugging time multiplies.                                                                                                            | 'Why add another agent when the Coordinator can handle it?' (short-term efficiency, long-term fragility).                                                                                                                              | Coordinator agent with domain-specific tools. No clear RACI between Coordinator and specialists.                                               | Time pressure ('just make it work') overrides architectural discipline.                                                                   | Refactor: remove ALL domain tools from the Coordinator. Create a new specialist for any task the Coordinator was doing directly. Verify the Coordinator only delegates.                                                                | When defining the Coordinator agent, explicitly set its tool list to EMPTY (or delegation-only). Test by removing all specialists — the Coordinator alone should produce NOTHING useful. |
| **EPS.P3: State must be explicit and structured**       | Every piece of context passed between agents must be a structured object — not raw text. Unstructured handoffs are the #1 production failure mode in multi-agent systems.                                                               | Structured state = Pydantic models, JSON schemas, or markdown with consistent headers. Each agent declares its input schema (what it needs) and output schema (what it produces). The Coordinator matches outputs to inputs. No free-text passing.                                                                         | Works when: Agent A's output schema matches Agent B's input schema exactly. The Coordinator validates the match before passing. If there's a mismatch, the workflow stops and reports the error instead of silently degrading.      | Rooted in: type systems (programming), contract-based design, API schema validation, data quality management.                                                                                 | Explicit schemas mean: errors are caught at the handoff point, not 3 agents downstream. Each agent's reasoning is traceable (structured fields, not opaque paragraphs). The User can inspect any intermediate state and understand exactly what each agent received and produced.                             | Contract-based design, type safety, schema validation, data lineage.                                               | Pydantic models (Python), JSON Schema, Agno's structured output support, schema validation libraries.                                                                                           | Culture of 'schema first, then code'. Output format defined before agent instructions are written. Integration tests that verify handoff schemas.                                          | Violated when: agents pass raw text between each other. Agent B tries to 'understand' Agent A's free-text output. Different runs produce different formats. The Coordinator can't reliably route.                 | Path of least resistance — raw text is easier to produce than structured output. The agent 'just writes a paragraph' instead of filling in schema fields.                                                        | Unstructured handoffs fail probabilistically: sometimes Agent B correctly parses the text, sometimes it misses a field, sometimes it hallucinates a field that wasn't there. Failures are intermittent and nearly impossible to reproduce.                                      | 'Natural language is the universal interface' (for humans, yes; for agent-to-agent communication, no — structured data is). 'The LLM will figure out the format' (LLMs are unreliable formatters without explicit schema constraints). | No schema definition for agent outputs. Using string parsing instead of structured extraction. No validation between handoffs.                 | Tight deadlines that push 'get it working now, structure it later'. No schema design phase before coding.                                 | Retrofit schemas: take one agent's raw text output, identify the 5-7 key fields, define a Pydantic model, add output schema constraints to the agent's instructions, validate the output matches the schema.                           | Define output schemas for your first 2 agents BEFORE writing their instructions. Schema design IS agent design.                                                                          |
| **EPS.P4: Build incrementally, validate continuously**  | Never design a 7-agent swarm on paper and try to build it all at once. Start with 1 agent. Get it working. Add a 2nd. Wire them. Validate. Then add a 3rd. Each addition is tested before the next begins.                              | Incremental building with continuous validation. Each phase produces a working artefact. Each artefact is tested against real data. Each test result is captured in the ILE. No phase is skipped. Gate = the artefact works as specified.                                                                                  | Works when: the User resists the temptation to design the full system upfront. Phase gates enforce sequence. The definition of 'done' for each phase is concrete and testable.                                                      | Rooted in: agile development, test-driven development, lean startup (build-measure-learn), complexity theory (emergent behaviour of multi-agent systems is unpredictable — test empirically). | Incremental building works because: multi-agent systems have combinatorial failure modes. 7 agents = 21 possible pairwise interactions. Building incrementally lets you isolate WHICH interaction is failing. Building all-at-once means failures come from everywhere simultaneously.                        | Agile, TDD, lean startup, complexity theory, empirical testing over theoretical design.                            | Git (version each phase), automated tests (validate each agent independently), integration tests (validate each handoff), ILE Session Log (track progress), EOP phase gates (enforce sequence). | Culture of 'working software over comprehensive design'. Permission to ship imperfect Phase 1 before designing Phase 4. Celebrate each working phase as a milestone.                       | Violated when: User designs the full 7-agent architecture on paper, then tries to build it all at once. First run fails everywhere. Can't isolate the problem. Abandons the approach.                             | Intellectual excitement — designing the full system is more FUN than building one boring agent. The vision is compelling; the implementation is tedious. The temptation to skip ahead is constant.               | All-at-once building produces a system with 21 potential failure points. The first test fails. The User can't tell if the Coordinator, Agent A, the handoff schema, Agent B, or the workflow sequencing is broken. Debugging takes longer than incremental building would have. | 'If I plan it perfectly, it will work first time' (multi-agent systems are non-deterministic; planning ≠ reality). 'This is too simple for me' (simple foundations enable complex systems).                                            | Architecture diagrams treated as substitutes for working code. Framework evaluations without hands-on testing.                                 | No incremental milestones defined. No phase gates. No celebration of Phase 1 completion. Only the final vision is considered 'success'.   | Break any stalled build into: what is the SMALLEST working thing I can build right now? Build that. Test it. Commit it. Then ask: what is the NEXT smallest working thing?                                                             | Define Phase 1 milestone: 'One agent, one tool, one structured output, tested against real data.' Don't think about Phase 2 until Phase 1 is done.                                       |
| **EPS.P5: The workflow is the product, not the agents** | Individual agents are commodities — any LLM can be a 'Research Analyst'. The WORKFLOW (which agent runs when, what feeds what, where the gates are, how errors are handled) is your competitive advantage and your OE engine's core IP. | The workflow defines: step sequencing (which agents run in what order), parallelism (which agents can run simultaneously), conditions (if X then Agent A, else Agent B), loops (retry on failure), error handling (what happens when an agent fails), and human-in-the-loop gates (where the Director reviews).            | Works when: the User invests 70% of design time in workflow architecture and 30% in agent prompts. Two people with the same Agno framework and the same Claude model produce completely different results based on workflow design. | Rooted in: process engineering, business process management (BPM), manufacturing (assembly line design > individual station design), orchestration pattern.                                   | Workflow-as-product works because: agents can be individually improved or replaced without changing the workflow. The workflow captures the User's THINKING about how the domain works (e.g., 'risk assessment MUST happen after deep dive but before memo writing'). This thinking is the OE engine's value. | Process engineering, BPM, assembly line design, orchestration pattern, value stream mapping.                       | Agno Workflow class (Loop, Parallel, Condition, Router), workflow-as-code (version controlled, testable), visual workflow monitoring (AgentOS dashboard).                                       | Culture of 'design the process, then staff it with agents'. Workflow reviewed and approved before agents are built. Process diagrams created before code.                                  | Violated when: the User focuses all effort on perfecting individual agent prompts while the workflow is ad-hoc. Agents produce great individual outputs but the assembly is chaotic.                              | Agent-centric thinking — it's natural to focus on making each agent 'smart'. But a team of brilliant individuals with no process produces worse results than a team of average individuals with a great process. | Brilliant agents + bad workflow = inconsistent outputs. Some runs produce great memos, others produce garbage, depending on which agent happened to run first or which context was passed. The system is unreliable.                                                            | 'Better agents = better system' (better PROCESS = better system). 'The agent's prompt is the most important thing' (the workflow sequencing is more important).                                                                        | Obsessive prompt engineering on individual agents while workflow architecture is undefined. A/B testing agent prompts without stable workflow. | No workflow documentation. No process diagram. No explicit step sequencing. Agents run in ad-hoc order decided by the Coordinator's mood. | Draw the workflow FIRST: boxes (agents) and arrows (data flow). Define: what runs in parallel? What runs sequentially? Where are the gates? Where are the error handlers? THEN build the agents to fit the workflow.                   | Before building Phase 3 (multi-agent workflow), draw the workflow diagram on paper. Show it to Claude for review. Get the process right before writing agent code.                       |
| **EPS.P6: Observability is not optional**               | If you can't see what your agents are doing (which tools they called, what they reasoned, where they failed), you can't debug, trust, or improve the system. Observability is the foundation of operational excellence.                 | Observability = the ability to understand the internal state of the system by examining its outputs. For AI orchestration: streaming events (real-time reasoning), tool call logs, inter-agent message logs, error traces, cost tracking, latency metrics, and audit trails.                                               | Works when: every agent run produces a traceable record. The User can answer: 'Why did the Risk Officer reject this thesis?' by reading the trace, not by guessing.                                                                 | Rooted in: observability engineering (distributed systems), DevOps, site reliability engineering (SRE), audit compliance.                                                                     | Observability works because: agents are non-deterministic. The same input produces different outputs on different runs. Without observability, you can't tell if a good output was skill or luck, or if a bad output was a prompt error, a tool failure, or a model hallucination.                            | Observability engineering, SRE, distributed tracing, audit compliance.                                             | Agno stream_events=True (real-time event streaming), AgentOS dashboard (visual tracing), structured logging (JSON logs per agent run), cost tracking per run.                                   | Culture of 'if it's not observable, it's not production'. Observability built from Day 1, not added later. Dashboard checked after every run.                                              | Violated when: agents run as black boxes. The User sees only the final output. When it's wrong, there's no trace to diagnose. Debugging becomes 'change the prompt and hope'.                                     | Observability feels like overhead — 'just make it work first, add logging later'. But without logging, 'make it work' becomes impossible because you can't see what's broken.                                    | Black-box agents create a debugging death spiral: output is wrong → change the prompt → output is different-wrong → change the prompt again → cycle repeats. Without observability, each change is a blind guess.                                                               | 'Logging slows things down' (negligible overhead). 'I'll add observability when it's in production' (you need observability MOST during development).                                                                                  | Running agents without stream_events=True. No structured logging. No AgentOS dashboard. No cost tracking.                                      | Time pressure ('just ship it'). Unfamiliarity with tracing tools. No observability habit in current workflow.                             | Add stream_events=True to the very first agent you build. Read ONE event trace end-to-end. Understand: model reasoning → tool selection → tool call → tool result → model response. This 10-minute investment saves hundreds of hours. | Set stream_events=True in your first agent (Phase 1, Step 1.3). Read the trace. Understand it before proceeding to Phase 2.                                                              |
