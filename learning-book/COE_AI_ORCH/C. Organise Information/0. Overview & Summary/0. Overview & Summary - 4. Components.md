<td><strong>C1: Agent Runtime (Agno Framework)</strong></td>
<td>The foundational execution layer — without it, nothing runs. It provides the primitives (Agent, Team, Workflow) that all other components build upon.</td>
<td>The software framework that defines, instantiates, and executes AI agents. Provides: Agent class (model + tools + instructions + memory), Team class (Coordinator + specialists), Workflow class (steps with control flow). Agno is the primary choice; CrewAI is the secondary/comparison.</td>
<td>Works when: agents are defined declaratively (~20 lines Python), instantiate in microseconds, execute tools reliably, stream results in real-time, and compose into teams without glue code.</td>
<td>Clean API design (Python-native, minimal boilerplate), model-agnostic architecture (swap LLMs without changing agent logic), built-in composition primitives (Teams, Workflows).</td>
<td>Clean API reduces code → fewer bugs → faster iteration. Model-agnostic means the User isn't locked to one LLM provider. Composition primitives mean teams are first-class, not hacked together.</td>
<td>EPS.P1 (specialists via Agent class), EPS.P2 (Coordinator Mode), EPS.P4 (incremental via single Agent → Team → Workflow).</td>
<td>pip install agno, Python 3.10+, Anthropic/OpenAI API keys, Cursor IDE for development.</td>
<td>Stable Python environment, reliable internet for API calls, sufficient API quota for development iteration.</td>
<td>Fails when: framework is too complex for the User's level (LangGraph, AutoGen). Or too simple for production needs (smolagents). Or has breaking API changes that destroy existing agents.</td>
<td>Framework mismatch — choosing a framework that doesn't match the User's skill level or use case. Too complex = never ships. Too simple = can't scale.</td>
<td>Complex frameworks require deep programming knowledge → User gets stuck on framework boilerplate instead of building agents. Simple frameworks lack composition primitives → User has to build their own team/workflow logic → bugs, delays.</td>
<td>'The most popular framework must be the best' (popularity ≠ fit). 'I should evaluate all frameworks before choosing' (analysis paralysis).</td>
<td>LangGraph (graph abstraction adds complexity), building from raw API calls (no framework primitives), using multiple frameworks simultaneously.</td>
<td>No commitment to one framework. Constant switching. No depth in any single tool.</td>
<td>Commit to Agno for L1-L3. Evaluate CrewAI at L4+ when you have production experience to compare against. Do NOT switch frameworks during foundational learning.</td>
<td>Install Agno. Build one agent. Don't evaluate alternatives until you've deployed a 4+ agent team.</td>
</tr>
<tr>
<td><strong>C2: LLM Provider (Claude / Model Layer)</strong></td>
<td>The intelligence layer — the 'brain' inside each agent. Without a capable LLM, agents can't reason, use tools, or produce quality outputs. The model choice affects reasoning quality, tool use reliability, cost, and latency.</td>
<td>The large language model that powers each agent's reasoning. Claude (Anthropic) is the primary choice for strong reasoning and tool use. Agno is model-agnostic, so different agents can use different models (e.g., Claude for analysis, Gemini for Committee Chair).</td>
<td>Works when: the LLM reliably follows agent instructions, correctly selects and uses tools, produces structured output matching the schema, and handles edge cases gracefully.</td>
<td>Claude's reasoning depth + tool use reliability. Model-agnostic framework means the User can swap models per-agent based on cost/capability tradeoffs.</td>
<td>Strong reasoning → higher-quality agent outputs → more trustworthy system → User relies on agents more → leverage increases. Model-agnostic → cost optimisation (use cheaper models for simple agents, powerful models for complex ones).</td>
<td>EPS.P1 (specialist agents may need different model capabilities). EPS.P6 (observability of model reasoning via streaming).</td>
<td>Anthropic API key, Claude API (claude-sonnet-4-5-20250929 or later), optional: OpenAI API for GPT, Google API for Gemini.</td>
<td>API cost budget (track per-agent, per-run), reliable API availability, rate limit awareness.</td>
<td>Fails when: model hallucinates tool calls, doesn't follow output schemas, or costs more than the value it produces.</td>
<td>Model over-reliance — assuming the LLM is always right. Not validating model outputs against schemas. Not monitoring cost per run.</td>
<td>Unvalidated model output propagates errors through the workflow. Agent A hallucinates a number → Agent B bases its analysis on the hallucination → final memo contains confident-sounding nonsense.</td>
<td>'AI doesn't make mistakes' (LLMs hallucinate regularly). 'Bigger model = always better' (bigger = more expensive; sometimes a smaller model suffices).</td>
<td>Using the most expensive model for all agents regardless of task complexity. No output validation. No cost tracking.</td>
<td>No API cost alerts. No output schema validation. No human review of critical agent outputs.</td>
<td>Add output schema validation to every agent. Use cheaper models for simple tasks (formatting, routing). Reserve expensive models for complex reasoning (analysis, decision-making). Track cost per run.</td>
<td>Set up API key. Configure cost tracking from Day 1. Start with Claude Sonnet for all agents, then optimise per-agent at L3+.</td>
</tr>
<tr>
<td><strong>C3: Development Environment (Cursor IDE)</strong></td>
<td>The workspace where agent code is written, tested, and iterated. The IDE choice directly affects development speed and learning velocity.</td>
<td>Cursor IDE with Claude AI integration. Provides: AI-assisted code generation (describe agent in English → Claude writes Python), inline debugging, terminal for running agents, Git integration. The ILE also lives here — learning and building in one workspace.</td>
<td>Works when: the User can describe an agent in natural language, Claude generates working code, the User runs it in the terminal, sees the output, understands it, and iterates — all within one window.</td>
<td>AI-assisted coding reduces the Python barrier. The User's existing Cursor proficiency (4 prototypes in 1 month) means the environment is already familiar.</td>
<td>Familiar environment → no 'new tool' overhead. AI code generation → Python barrier is dramatically lower. Integrated terminal → immediate feedback loop. ILE in same workspace → zero-context-switch learning.</td>
<td>EPS.P4 (incremental: write, run, test, iterate in tight cycles).</td>
<td>Cursor IDE, Claude integration (API key), Python extension, Git extension, terminal.</td>
<td>Quiet focused work sessions, fast internet for AI code generation, multiple monitor setup (code + agent output + ILE simultaneously).</td>
<td>Fails when: the User relies entirely on Claude-generated code without understanding it. Or uses Cursor as a crutch instead of learning Python fundamentals.</td>
<td>Black-box coding — Claude writes all the code, User runs it but can't modify or debug it. The User becomes dependent on Claude for every change.</td>
<td>Black-box dependency means: any non-trivial bug requires Claude to fix. If Claude's suggestion is wrong, the User can't identify the error. Development velocity drops to zero when Claude is unavailable.</td>
<td>'I don't need to understand the code, just the architecture' (architecture without implementation understanding = fragile knowledge).</td>
<td>Auto-accepting all Claude suggestions without reading them. No manual code writing practice. No debugging practice.</td>
<td>Always-accept-suggestion workflow. No time spent understanding generated code.</td>
<td>For every Claude-generated agent: read each line, add a comment explaining what it does, modify one thing manually, run it, verify the modification worked. This builds understanding alongside speed.</td>
<td>Use Cursor for all agent development. But commit to understanding EVERY line of your first 3 agents by adding inline comments.</td>
</tr>
<tr>
<td><strong>C4: Monitoring & Operations (AgentOS)</strong></td>
<td>The operational visibility layer — the Director's dashboard. Without it, the User is directing blind.</td>
<td>AgentOS: Agno's pre-built FastAPI app and web UI for real-time orchestration, debugging, and deployment. Provides: per-user/session isolation, runtime approval enforcement, native tracing, full auditability, cost tracking.</td>
<td>Works when: the User can open a browser, see all running agents, trigger runs, view traces, identify failures, and measure costs — without touching code.</td>
<td>Built-in to Agno (no separate install). Visual interface (non-technical Director can monitor). Real-time streaming of agent activity.</td>
<td>Visual monitoring → the User trusts the system because they can SEE it working. Tracing → failures are diagnosed in minutes, not hours. Cost tracking → no surprise bills.</td>
<td>EPS.P6 (observability is not optional). This component IS the operationalisation of that principle.</td>
<td>AgentOS (ships with Agno), web browser, FastAPI server, database for traces/sessions.</td>
<td>Server to run AgentOS (local or cloud), persistent storage for traces, network access from User's browser.</td>
<td>Fails when: AgentOS is not set up (the User runs agents from terminal only). Or the dashboard is ignored ('I'll check it later').</td>
<td>Terminal-only habit — the User is comfortable running agents from Cursor terminal and never opens the dashboard. Observability exists but isn't used.</td>
<td>Running from terminal means: traces are ephemeral (scroll off screen), costs aren't tracked (no dashboard metrics), failures aren't recorded (no audit trail). The User operates blind.</td>
<td>'Terminal output is enough' (for development, maybe; for operations, no). 'I'll set up monitoring when it's in production' (by then it's too late).</td>
<td>Terminal-only execution without logging. No AgentOS server running. No trace storage.</td>
<td>No server infrastructure. Not knowing how to set up AgentOS. Treating monitoring as 'later' work.</td>
<td>Set up AgentOS in Phase 3 (deployment). Make it the FIRST thing you do in that phase — before running the full workflow. Open it in a browser tab that stays open during all development.</td>
<td>Include AgentOS setup as Phase 3, Step 3.3 in the EOP. It's a prerequisite for multi-agent deployment.</td>
</tr>
<tr>
<td><strong>C5: Knowledge Persistence (ILE + Agent Memory)</strong></td>
<td>The memory layer — what the User learns AND what the agents remember across sessions. Without persistence, every session starts from zero.</td>
<td>Two sides: (1) User-side: ILE Learning Book captures the User's learning (decomposition patterns, workflow designs, failure post-mortems). (2) Agent-side: session memory (Agno storage), knowledge bases (vector DBs), knowledge graphs. The ILE is the User's brain; agent memory is the system's brain.</td>
<td>Works when: the User resumes after a week and the ILE tells them exactly where they left off (Subject Roadmap A). Agents resume after a restart and remember previous analysis (session memory, knowledge base).</td>
<td>ILE framework already exists and is proven (COE_DS as evidence). Agno provides session memory and knowledge base integration out of the box.</td>
<td>User-side persistence → no re-learning, no re-explaining, no lost context. Agent-side persistence → agents build on prior work instead of starting fresh each run.</td>
<td>EPS.P3 (state must be explicit — memory is explicit state). The ILE's 'hierarchy over chronology' principle applies to agent memory too.</td>
<td>ILE Learning Book (markdown, Git-tracked), Agno session memory, vector databases (for RAG), knowledge graphs (for relationship mapping), Git (version control for all persistent state).</td>
<td>Consistent session logging discipline, database infrastructure for agent memory at L3+, Git commit habit after every working phase.</td>
<td>Fails when: the User doesn't update the ILE (learning is volatile). Or agents have no memory configured (every run is stateless).</td>
<td>Documentation tax — updating the ILE and configuring agent memory feels like overhead during the 'building' phase. The User wants to build, not document.</td>
<td>Skipped documentation means: the User forgets why they made design decisions. Agents repeat prior analysis instead of building on it. When the User resumes after a break, they spend 2 hours re-establishing context that should have taken 2 minutes.</td>
<td>'I'll remember why I did it this way' (you won't). 'Stateless is simpler' (stateless means every run ignores all previous work).</td>
<td>No ILE updates. No Git commits. No session memory configured. No knowledge base.</td>
<td>Building sessions without ILE checkpoints. No discipline for post-session documentation.</td>
<td>Adopt the ILE's existing session log habit: at the END of every work session, update A (Session Log) with: what you worked on, what you learned, what's next. This takes 5 minutes and saves hours.</td>
<td>Configure ILE session logging for AI Orchestration subject (A is already created). Add session memory to agents starting in Phase 2.</td>
</tr>
<tr>
<td><strong>C6: Version Control (Git/GitHub)</strong></td>
<td>The reproducibility and rollback layer. Without it, changes are irreversible, collaboration is impossible, and the system can't be reproduced on another machine.</td>
<td>Git for all agent code, prompts, workflow definitions, configuration, and documentation. GitHub for remote storage, collaboration, and CI/CD at L4+. Every working phase is a commit. Every breaking change can be rolled back.</td>
<td>Works when: any state of the system can be recreated from a Git checkout. The User can rollback to yesterday's working version in seconds. Agent prompt changes are tracked with diffs.</td>
<td>Git is ubiquitous, free, and the User already uses it (ILE is Git-tracked). GitHub adds remote backup, collaboration, and action automation.</td>
<td>Version tracking → the User can see WHAT changed between a working version and a broken version. Rollback → breaking changes are recoverable. History → all design decisions are traceable.</td>
<td>EPS.P4 (build incrementally — each increment is a commit). EPS.P3 (explicit state — Git IS the explicit state of the system).</td>
<td>Git, GitHub, Cursor Git integration, .gitignore (exclude API keys, .env files).</td>
<td>Commit discipline (commit after every working phase), branch strategy (main = working, feature branches for experiments), .env for secrets (never committed).</td>
<td>Fails when: the User doesn't commit regularly. Or commits API keys. Or doesn't use branches (all changes go to main, no rollback possible).</td>
<td>Commit discipline gap — the User builds for hours without committing. A breaking change destroys hours of work with no rollback.</td>
<td>No commits means: breaking changes are permanent. The User spends hours recreating a state that existed 30 minutes ago. Lost work kills motivation.</td>
<td>'I'll commit when it's done' (commit when it WORKS, even partially). 'Git is for teams' (Git is for ANYONE who values their work).</td>
<td>No commit habit. No .gitignore. API keys in code instead of .env files.</td>
<td>No Git workflow established. Building without version control.</td>
<td>Commit after every working test. Use descriptive commit messages: 'Phase 1: single Finance Agent with YFinance, structured output verified.' Branch for experiments: 'experiment/add-risk-agent'.</td>
<td>Set up Git repo before writing any agent code. First commit: empty project structure + .gitignore + README. Build the habit from line 1.</td>
</tr>
<tr>
<td><strong>C7: Work Management (ClickUp via Cowork MCP)</strong></td>
<td>The operational tracking layer — connects the OE engine to the User's existing single source of truth for tasks, deadlines, and deliverables.</td>
<td>ClickUp connected via Claude Cowork MCP connector. At L1-L3: manual task tracking (User updates ClickUp with learning/build progress). At L4+: agents create/update ClickUp tasks autonomously (investment memo completed → ClickUp task marked done).</td>
<td>Works when: every learning milestone and build phase has a corresponding ClickUp task. Progress is visible. At L4+, agent outputs trigger ClickUp updates without human intervention.</td>
<td>User already uses ClickUp as SSOT. Claude Cowork MCP connector enables direct integration. The ILE's coe-map.yaml provides the mapping structure.</td>
<td>ClickUp integration creates accountability (deadlines), visibility (progress tracking), and at L4+ automation (agents update tasks). The Director sees the full operation in one dashboard.</td>
<td>EPS.P6 (observability extends to operations, not just agent internals). OE principle: SSOT for all operational state.</td>
<td>ClickUp account, Claude Cowork MCP connector, coe-map.yaml (ILE mapping config), ClickUp API (for L4+ agent integration).</td>
<td>ClickUp workspace with AI Orchestration space/project structure. Cowork connected. Task templates for learning milestones and build phases.</td>
<td>Fails when: ClickUp tracking is abandoned ('I'll update it later'). Or agent-ClickUp integration is attempted too early (before manual workflow is proven).</td>
<td>Tracking overhead — maintaining ClickUp feels like admin work that slows building. The User prefers to 'just build' and track later.</td>
<td>Without tracking: deadlines slip invisibly. The User can't measure velocity (how fast they're progressing). Stakeholders (if any) have no visibility. The User themselves can't see how far they've come.</td>
<td>'I know where I am without tracking' (you don't — cognitive load increases with system complexity). 'ClickUp is for teams, not solo operators' (solo operators need accountability MORE, not less).</td>
<td>No ClickUp tasks for learning milestones. No progress updates. No velocity measurement.</td>
<td>No ClickUp structure for AI Orchestration project. Cowork not connected to ClickUp.</td>
<td>Create a ClickUp project for AI Orchestration. Add tasks for each EOP phase and step. Update status after each session. This takes 5 minutes and provides weeks of visibility.</td>
<td>Connect ClickUp via Cowork MCP (connector available). Create tasks for EOP Phase 1 steps. Track from Day 1.</td>
</tr>
<tr>
<td><strong>C8: Domain Tool Layer (pluggable per UE project)</strong></td>
<td>The domain-specific capability layer — what makes agents useful for a specific domain. This is the ONLY component that changes when the OE engine is repointed to a new domain.</td>
<td>Pluggable tool integrations that agents use to interact with domain data. For Investment: YFinance (financial data), Exa (web search). For Research: Exa, academic databases. For Operations: ClickUp API, internal APIs. The OE engine is domain-agnostic; this layer is domain-specific.</td>
<td>Works when: tools are well-documented, produce structured output, handle errors gracefully, and can be swapped in/out without changing agent or workflow architecture.</td>
<td>Agno's 100+ tool integrations provide pre-built connectors. The plugin pattern means adding a new domain = adding new tools, not rebuilding the system.</td>
<td>Pluggable tools mean: the same analyst agent template with different tools becomes a Financial Analyst (YFinance) or a Research Analyst (Exa + academic DBs) or an Operations Analyst (ClickUp API). Architecture stays constant.</td>
<td>EPS.P1 (specialist agents + specialist tools). EPS.P5 (workflow stays constant; only tools change per domain).</td>
<td>Agno tool integrations (YFinance, Exa, web search, file I/O, etc.), custom tool definitions (Python functions wrapped as Agno tools), domain APIs.</td>
<td>Access to domain data sources, API keys for domain services, domain expertise (or access to domain experts) for validating tool outputs.</td>
<td>Fails when: tools produce unreliable data (garbage in → garbage out). Or tools are tightly coupled to agent logic instead of cleanly pluggable.</td>
<td>Tool reliability assumption — the User assumes tool outputs are always correct. Financial data has errors. Web search returns irrelevant results. APIs go down.</td>
<td>Unreliable tools produce wrong intermediate data → agents reason on wrong data → final output is confident-sounding nonsense. The User trusts the output because 'the agent did the research'.</td>
<td>'Data from APIs is always accurate' (it's not — APIs have bugs, delays, missing data). 'If the agent used a tool, the result must be right' (tool results need validation).</td>
<td>Unvalidated tool outputs. No error handling for API failures. No data quality checks.</td>
<td>No test data to validate tool outputs against. No error handling patterns. No fallback data sources.</td>
<td>Add data validation to every tool call: check for null/missing values, validate ranges (stock price > 0), cross-reference critical data with a second source. Build tool error handling in Phase 3.</td>
<td>In Phase 1, use YFinance and verify the output manually (check price against Google Finance). Build the validation habit before automating.</td>
</tr>
