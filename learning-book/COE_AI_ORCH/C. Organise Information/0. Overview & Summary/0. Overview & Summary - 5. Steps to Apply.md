<td><strong>PHASE 1: Single Agent Mastery (L1 — Follow, Weeks 1-2)</strong></td>
<td>Foundation phase. Proves the User can build, run, and observe ONE agent with real tools. Without this, all subsequent phases fail. This phase closes the 'can I make an agent?' question definitively.</td>
<td>Build a single Finance Agent using Agno: model=Claude, tools=[YFinance], instructions='Senior Finance Analyst', structured output. Run with stream_events=True. Observe tool calls. Modify and re-run. Commit to Git. Capture in ILE.</td>
<td>Steps: (1) Install Agno + Python environment. (2) Build first agent (~20 lines). (3) Run with streaming, observe events. (4) Modify (change ticker, change output format). (5) Add structured output schema. (6) Commit. (7) Update ILE Session Log.</td>
<td>Cursor writes 80% of code. Agno quickstart docs provide working examples. The User's 4 prototype experiences mean the dev environment is already familiar.</td>
<td>Fast iteration cycle: describe → generate → run → observe → modify → run again. Each cycle takes minutes, not hours. Confidence builds through direct experience, not theory.</td>
<td>EPS.P4 (build incrementally — this IS the increment). EPS.P6 (observability from Day 1 via streaming). EPS.P1 (one agent, one job, from the start).</td>
<td>Python 3.10+, pip install agno, Cursor IDE, Anthropic API key, YFinance tool, Git repo.</td>
<td>2-hour focused sessions. Working internet. No distractions. Permission to build something 'simple'.</td>
<td>Fails when: User skips to Phase 2 ('single agent is too easy'). Or doesn't add streaming ('I'll add observability later'). Or doesn't commit to Git ('it's just a test').</td>
<td>Underestimation — 'I already know how to talk to Claude, single agent is beneath me.' This skips the critical translation from chatbot-thinking to agent-thinking.</td>
<td>Skipping Phase 1 means the User enters Phase 2 without understanding tool calls, streaming events, or structured output. Phase 2 failures are then impossible to debug because Phase 1 skills are missing.</td>
<td>'I can skip foundations because I'm experienced with AI' (chatbot experience ≠ agent experience).</td>
<td>No structured output schema. No streaming. No Git commit.</td>
<td>Impatience. Identity as 'advanced user'. No defined Phase 1 milestone.</td>
<td>Define a concrete gate: 'My agent returns live AAPL price via YFinance tool call (not hallucinated), I can read the stream trace, and I can explain the difference between a chatbot and an agent.' Don't proceed until YES.</td>
<td>Start NOW. Open Cursor. pip install agno. Write 20 lines. Run it. See it work. Commit.</td>
</tr>
<tr>
<td><strong>PHASE 2: Two-Agent Team (L2 — Assist, Weeks 3-4)</strong></td>
<td>Coordination phase. Proves the User can make TWO agents collaborate through a Coordinator with structured handoffs and session memory. This phase introduces the core orchestration pattern.</td>
<td>Build a 2-agent Team: Research Analyst + Risk Analyst with Coordinator Mode. Research Analyst outputs structured thesis → Coordinator passes to Risk Analyst → Risk Analyst challenges and produces revised assessment. Session memory persists context.</td>
<td>Steps: (1) Design roles on paper (job description + I/O schema per agent). (2) Build Team with Coordinator Mode. (3) Test handoff (verify Research output feeds Risk input). (4) Add session memory. (5) Test follow-up question. (6) Commit. (7) Update ILE.</td>
<td>Phase 1 skills (agent building, streaming, structured output) transfer directly. Coordinator Mode is built into Agno. The User only needs to learn Team composition.</td>
<td>Phase 1 confidence + Agno's Team abstraction make this a natural extension: 'I built one agent; now I'm connecting two.' The cognitive load is bounded because each agent is already understood individually.</td>
<td>EPS.P2 (Coordinator directs, never does). EPS.P3 (structured state between agents). EPS.P4 (incremental — 1 agent → 2 agents, not 1 → 7).</td>
<td>Agno Team class, Coordinator Mode, Pydantic models for I/O schemas, session memory storage.</td>
<td>Phase 1 completed and committed. Quiet focused session. Paper/whiteboard for role design. Real test data.</td>
<td>Fails when: handoffs are unstructured (raw text between agents). Or the Coordinator also does domain work (violates EPS.P2). Or session memory isn't tested.</td>
<td>Handoff fragility — the Research Analyst's output doesn't match what the Risk Analyst expects. The Coordinator passes garbage. No schema validation catches it.</td>
<td>Unvalidated handoffs produce intermittent failures: some runs work (output happens to be parseable), some fail (output format varies). The User can't tell if the Research Analyst or the handoff is broken.</td>
<td>'The agents will figure out the format between them' (they won't — schemas must be explicit).</td>
<td>No Pydantic models for agent outputs. No schema validation at handoff. Coordinator with domain tools.</td>
<td>Rushing to 'get it working' without designing I/O schemas first.</td>
<td>Schema first: define the exact JSON/Pydantic model that Research Analyst outputs and Risk Analyst expects. THEN build the agents to produce/consume those schemas. THEN wire the handoff.</td>
<td>Before writing code: draw the handoff diagram. Define the schema. THEN build.</td>
</tr>
<tr>
<td><strong>PHASE 3: Multi-Agent Workflow + Deployment (L3 — Apply, Weeks 5-8)</strong></td>
<td>Architecture phase. Builds the full agent team (6+ agents) with structured workflow (Parallel, Condition, Loop) and deploys to AgentOS. This is the first 'real' OE engine instance.</td>
<td>Build a multi-agent workflow (e.g., based on Agno Investment Team template): 6+ specialist agents, workflow with Parallel (independent analyses), Condition (confidence-based routing), error handling. Deploy to AgentOS. Monitor via dashboard.</td>
<td>Steps: (1) Study Agno Investment Team template. (2) Map each agent to role/tools/schema. (3) Build workflow (draw first, code second). (4) Test end-to-end with real data. (5) Handle error scenarios. (6) Deploy to AgentOS. (7) Monitor and iterate. (8) Commit + ILE.</td>
<td>Phase 1-2 skills (agent building, team coordination, structured handoffs) provide the foundation. Agno Investment Team template provides the reference architecture. AgentOS provides deployment infrastructure.</td>
<td>The template accelerates: the User doesn't design from scratch but studies a working 7-agent system, understands it, then adapts it. This is 'standing on the shoulders of giants' — the hardest architecture decisions are already made.</td>
<td>EPS.P5 (workflow is the product — this phase IS building the workflow). EPS.P6 (observability via AgentOS). EPS.P4 (build the workflow incrementally — 2 agents first, then add the 3rd, etc.).</td>
<td>Agno Workflow class (Parallel, Condition, Loop, Router), AgentOS (deployment + monitoring), full tool suite for chosen domain, Git branches for workflow experiments.</td>
<td>Phase 2 completed. Access to reference template (Agno Investment Team). Real domain data for end-to-end testing. Time for iterative workflow refinement (this phase takes 4 weeks).</td>
<td>Fails when: the User builds all 6+ agents simultaneously without testing incrementally. Or skips AgentOS deployment ('terminal is fine'). Or doesn't test error scenarios.</td>
<td>All-at-once building — designing 7 agents and a complex workflow on paper, then building it all at once. First run produces 21 potential failure points.</td>
<td>21 possible pairwise interactions in a 7-agent system. One failing handoff cascades through the workflow. Without incremental testing, the User can't isolate which handoff is broken. Debugging becomes combinatorial.</td>
<td>'The template works, so my version will too' (templates are starting points, not guarantees — your domain and data differ).</td>
<td>Building all agents before testing any. No error handling. No AgentOS deployment. No incremental testing.</td>
<td>Over-reliance on template without understanding. No workflow testing harness. No error scenario planning.</td>
<td>Build in pairs: (a) Market + Financial analysts as a 2-agent team. Test. (b) Add Technical analyst. Test. (c) Add Risk. Test. (d) Add Writer + Chair. Test. Each addition = one commit. Never add 2 agents simultaneously.</td>
<td>Clone the Agno Investment Team repo. Read every file. Map each agent to its role. Then rebuild from scratch (don't just copy) — understanding requires building, not reading.</td>
</tr>
<tr>
<td><strong>PHASE 4: Multi-Domain Reuse + Production Governance (L4-L5 — Enable/Ensure, Weeks 9-16)</strong></td>
<td>OE Engine proof phase. Proves the architecture is composable (repoints to a second domain) and operable (governance, cost control, standards). This is when the User transitions from Builder to Director.</td>
<td>Extract reusable patterns from Phase 3 into base templates. Deploy a second domain team using the same architecture. Establish production governance: cost tracking, error budgets, security review, written standards. Evaluate CrewAI as comparison.</td>
<td>Steps: (1) Refactor Phase 3 into base templates (base analyst, base writer, base coordinator). (2) Build Team 2 for second domain using templates. (3) Establish governance (cost dashboard, error budgets). (4) Write standards doc. (5) Evaluate CrewAI with same workflow. (6) Integrate with ClickUp. (7) Update ILE with full system documentation.</td>
<td>Phase 3 working system provides the foundation to extract patterns from. Template extraction is a refactoring exercise (the User already has working code to refactor). Governance tools are built into AgentOS.</td>
<td>The transition from Builder to Director happens when: the User spends more time DESIGNING workflows and REVIEWING outputs than WRITING code. Templates make this possible — new teams are composed, not built from scratch.</td>
<td>EPS.P5 (workflow is the product — templates capture the workflow). All principles converge in this phase: specialists (P1), separation (P2), structured state (P3), incremental (P4), workflow-centric (P5), observable (P6).</td>
<td>Full Agno suite, base template library (created in this phase), second domain's tool integrations, CrewAI (for comparison), ClickUp MCP connector, governance dashboard.</td>
<td>Phase 3 completed and stable. Access to second domain data. Time allocated for refactoring (not just building). Willingness to establish standards (discipline over speed).</td>
<td>Fails when: templates are too domain-specific (Phase 3 code has hard-coded domain logic). Or governance is skipped ('I'll add monitoring later'). Or CrewAI evaluation is skipped ('Agno works, why bother?').</td>
<td>Template extraction resistance — the User wants to keep building new things instead of refactoring old things. Extracting reusable patterns feels like 'going backwards'.</td>
<td>Without template extraction, Team 2 is built from scratch. Time-to-deploy doesn't improve. The compound returns (UDS.UD) never materialise. The OE engine promise is broken.</td>
<td>'I'll make it reusable later' (later = never). 'This domain is different' (the orchestration pattern is the SAME; only tools and prompts differ).</td>
<td>Domain-specific hard-coding. No template library. No governance. No framework comparison.</td>
<td>No refactoring time allocated. No standards document. No governance dashboard. Building new things is more exciting than refactoring.</td>
<td>Block 1 week SPECIFICALLY for template extraction before starting Team 2. Define: what is the base analyst? What is the base coordinator? What is the base workflow? Abstract these into parameterised templates.</td>
<td>After Phase 3 is stable: spend Week 9 refactoring. Week 10: build Team 2 from templates. Weeks 11-12: governance. Weeks 13-14: CrewAI evaluation. Weeks 15-16: ClickUp integration + standards doc.</td>
