<td><strong>UBS.UB: No mental model for agent task decomposition</strong></td>
<td>This is the root causal driver of the UBS. Without a mental model for how to split a task into independent agent roles, the User cannot architect multi-agent systems. This blocker is NOT a knowledge deficit (the User knows what agents are) but a pattern recognition gap—the User hasn't internalized HOW to decide "this should be Agent A, that should be Agent B." Understanding this blocker is critical because tooling and frameworks alone cannot overcome a conceptual gap.</td>
<td>The blocker is the absence of a schema for task decomposition. It manifests as: User sees a complex task as "one monolithic AI system" rather than "5 specialist agents with handoff contracts." When prompted to decompose, the User struggles to explain WHY agents should be divided that way. The division feels arbitrary rather than principled. This is not a coding problem—it's a cognitive structure problem.</td>
<td>Success occurs when the User develops a repeatable decomposition schema: (1) identify the domain specialties required (research, analysis, synthesis, validation, communication), (2) map each specialty to an agent role with explicit input/output, (3) define the handoff sequence (which agent calls which, in what order), (4) validate that each agent owns a coherent scope (no overlap, no missing dependencies), (5) document the logic so other analysts/operators can understand the decomposition.</td>
<td>The driver overcoming this blocker is the ILE framework itself. Each chapter/topic in the ILE embodies a cognitive role—e.g., "Understand the Problem" (analyst role), "Translate to Task" (architect role), "Implement" (developer role). By mapping these ILE structures to agent scopes, the User converts abstract decomposition into a concrete, learnable pattern. The ILE becomes a decomposition scaffold.</td>
<td>The mechanism works as follows: User reads an ILE chapter and recognizes it as a specialist cognitive role. User then asks: "What specialist agent would own THIS cognitive task?" By translating ILE chapters to agent roles, the User practices decomposition repeatedly. The mental model builds through pattern recognition—"oh, these 5 chapters = these 5 agents." After 2-3 cycles, decomposition becomes intuitive, not effortful.</td>
<td>Underlying principles: (1) decomposition is learnable through pattern recognition (not innate), (2) explicit scoping prevents overlap (agents must own distinct, named responsibilities), (3) handoff contracts create boundaries (what does Agent A output that Agent B consumes?), (4) the ILE structure itself is a decomposition template (chapters = specialist roles), (5) decomposition must be documented so intent transfers to other operators.</td>
<td>Tools enabling success: (1) ILE Learning Book (provides decomposition templates), (2) Agno Investment Team reference architecture (working example of 7-agent decomposition), (3) Pydantic for schema definition (makes I/O contracts explicit), (4) Cursor for rapid prototyping (fast enough to iterate on decomposition decisions), (5) Git (captures decomposition history for learning reflection).</td>
<td>Environmental enablers: (1) access to domain experts who can validate decomposition logic, (2) permission to iterate decomposition without penalty (fail safe), (3) reference architectures for similar domains (short-circuits design by analogy), (4) learning cohort or peer review (decomposition validated externally, not just by the User), (5) time for reflection between builds (consolidation of pattern recognition).</td>
<td>Failure manifests as: (1) User builds multi-agent system but agents are poorly scoped (one agent handling both domain research AND synthesis—mixing roles), (2) agents overlap in responsibility (two agents doing similar work, creating inefficiency and confusion), (3) handoffs fail when task changes (agent scoping was task-specific, not generalizable), (4) User cannot explain decomposition logic to another analyst/operator (the scoping was implicit, felt rather than principled).</td>
<td>The blocker: User has no internalized pattern for deciding agent boundaries. When faced with a complex task, the User treats it as "one big problem" rather than "a sequence of specialist roles." The mental model is not yet developed. No amount of tooling can overcome this—the User must practice decomposition until it becomes automatic.</td>
<td>Mechanism of failure: User codes first, decomposes later (or not at all). The resulting agents are organized by implementation convenience, not by logical role boundaries. When another analyst/operator tries to extend the system, they cannot understand why agents are divided that way. The lack of decomposition reasoning means the system is not handoff-able. Each new task repeats this problem.</td>
<td>Risky principles: (1) "decomposition is obvious" (it's not—it requires practice), (2) "let the code reveal the structure" (implicit structures are not transferable), (3) "more agents = better" (without scoping discipline, more agents add confusion), (4) "I'll explain it in comments" (reasoning must be captured in architecture, not code comments), (5) "this is a one-off system" (prevents pattern generalization).</td>
<td>Risky tools: (1) LLM prompts that generate all agents at once ("generate 10 agents for this task")—prevents the User from learning decomposition, (2) multi-agent frameworks without templates (Agno without Investment Team reference), (3) code-first development (agent boundaries emerge from convenience, not design), (4) lack of schema enforcement (agents can be loosely defined, hiding decomposition gaps).</td>
<td>Risky environments: (1) isolated development (no peer review of decomposition logic), (2) pressure to ship prototypes (architecture deferred for speed), (3) treating prototypes as throwaway (decomposition learning not captured), (4) no reference architectures or patterns shared (User must invent decomposition from scratch), (5) ILE book used for learning only, not as active decomposition scaffold.</td>
<td>If the User is unable to decompose a task without guidance: (1) do NOT start coding, (2) map the task to ILE chapters—which ILE chapter would own each part of this task? (3) for each ILE chapter you identified, name an agent role (e.g., "Research Agent" maps to "Understand the Problem"), (4) write out the sequence: "Research Agent → Analysis Agent → Synthesis Agent," (5) for each arrow, write 1-2 lines describing the handoff (what does the first agent output that the second agent consumes?), (6) only then open Cursor and code.</td>
<td>Immediate next steps: (1) choose ONE of your 4 completed prototypes that felt most "coherent" in its agent scoping, (2) create a new ILE section titled "Decomposition Pattern: [Task Name]" and document that prototype's agent roles using the template: Agent Name | ILE Chapter Owner | Input Schema | Output Schema | Downstream Consumer, (3) identify the second-most-coherent prototype and repeat the documentation, (4) compare the two documented patterns—what's similar? that's your emerging decomposition schema, (5) for a NEW task, attempt decomposition using that emergent schema (test if pattern transfers).</td>

<td><strong>UBS.UB.UB: No structured I/O schema practice — User doesn't think in input→process→output for each specialist</strong></td>
<td>This is the deepest causal root of the decomposition blocker. Even if the User knows agents should be divided, without thinking in structured I/O contracts, agent boundaries collapse. An agent without an explicit input schema will accept any ambiguous input (mixing concerns). Without an explicit output schema, the downstream agent cannot rely on what it receives. This blocker prevents reusability because without contracts, agents are brittle.</td>
<td>The blocker is the absence of structured thinking about agent I/O. The User treats agents as natural language functions: "give the agent a task description, get back a response." This works for single-turn interactions but breaks in multi-agent orchestrations because: (1) input ambiguity causes the agent to hallucinate missing context, (2) output variability means downstream agents cannot parse responses reliably, (3) no schema means the User is not forced to think about what data truly belongs to each agent.</td>
<td>Success occurs when the User defines explicit Pydantic schemas for every agent I/O: (1) input schema specifies EXACTLY what data each agent needs (no more, no less), (2) output schema specifies EXACTLY what the agent produces (field names, types, constraints), (3) these schemas are enforced (agent code validates against schemas), (4) downstream agents depend on schema fields (not on natural language interpretation), (5) when the task changes, the schemas guide what must be updated.</td>
<td>The driver overcoming this blocker is: (1) mandatory use of Pydantic for agent I/O (User cannot build agents without defining schemas), (2) Cursor's ability to generate schema code from English descriptions (reducing Python friction), (3) reference patterns in Agno Investment Team showing how schemas are used in real agents, (4) deliberate exercises decomposing ILE sections into input→process→output patterns (schema thinking becomes habitual).</td>
<td>The mechanism: When the User must write a Pydantic schema, they are forced to answer explicit questions: "What fields does this agent actually need as input? Why?" This discipline surfaces implicit assumptions. The same happens with output: "What fields must the next agent receive? What type? What constraints?" By writing schemas repeatedly, the User internalizes I/O contract thinking. After 3-4 schema exercises, decomposing in terms of contracts becomes automatic.</td>
<td>Underlying principles: (1) I/O contracts are cognitive boundaries (they force clarity about scope), (2) schemas enforce reusability (downstream agents depend on contract, not on implementation details), (3) Pydantic validation prevents silent failures (missing fields raise errors, not just ignored), (4) explicit schemas enable testing (you can validate "does agent X produce the output schema it claims?"), (5) schemas document intent (other operators can read the contract and understand what the agent owns).</td>
<td>Tools: (1) Pydantic (enforces schema-based thinking), (2) Cursor code generation for schema boilerplate (reduces friction), (3) Agno agent_define templates showing schema patterns, (4) type hints in Python (documentation and validation in one), (5) testing frameworks that validate schema compliance, (6) ILE as schema repository (capturing patterns for reuse).</td>
<td>Environmental enablers: (1) learning partner who reviews schema definitions (validates that schemas are clean), (2) access to working reference schemas (Agno Investment Team shows real examples), (3) deliberate practice time (iterations on schema design without shipping pressure), (4) permission to fail (schemas that are not quite right after first iteration are learning artifacts), (5) tooling that makes schema changes easy (refactoring support).</td>
<td>Failure manifests as: (1) agents accept loose, untyped inputs (hallucination due to ambiguity), (2) agents output variable responses (one run produces {"result": "..."}, another produces {"output": "...", "reasoning": "..."}), (3) downstream agents brittle (they guess the field names in upstream responses), (4) debugging failures hidden (is the agent wrong, or is the downstream agent parsing wrong?), (5) code cannot be tested independently (no schema validation means no clear pass/fail).</td>
<td>The blocker: User has not internalized the discipline of explicit I/O contracts. Agents are defined by their instructions and examples, not by their data contracts. The User treats I/O as "whatever natural language emerges" rather than "a formalized schema."</td>
<td>Failure mechanism: Without schema discipline, agents are tightly coupled. Each agent must "guess" what the previous agent produces. Handoffs are fragile—a small change in Agent A's output breaks Agent B's parsing. This brittleness prevents reusability. Each new task requires re-engineering the handoffs. Schema-less decomposition is not reusable decomposition.</td>
<td>Risky principles: (1) "the LLM will figure out the format" (it won't reliably), (2) "I'll handle format mismatches in the calling code" (this creates brittle glue, not clean handoffs), (3) "schema is extra work" (it's actually a time-saver through reliability), (4) "type hints are optional" (they're cognitive anchors for intent), (5) "Pydantic is for data pipelines, not agents" (Pydantic is for any structured I/O).</td>
<td>Risky tools: (1) pure Python function calls without type hints (no enforcement), (2) string-based parsing ("find the result in the response"), (3) LLM-generated output without validation ("just ask the LLM to return JSON"), (4) frameworks that allow untyped agent responses (Agno can be misconfigured this way), (5) skipping Pydantic and using Python dict (lose validation and clarity).</td>
<td>Risky environments: (1) no code review of agent I/O design, (2) pressure to launch agents quickly (schema design skipped for speed), (3) agents built in isolation (no requirement to document handoffs), (4) no testing infrastructure (schema violations not caught until runtime), (5) learning book not consulted for I/O schema patterns.</td>
<td>If an agent's output is inconsistent or downstream agents fail to parse it: (1) do not patch the downstream parsing code, (2) stop and write an explicit output schema for the upstream agent, (3) add Pydantic validation in the agent code, (4) test the agent in isolation (does it produce valid schema?), (5) now revisit the downstream agent—if the upstream schema is clean, downstream parsing is simple, (6) document the schema in the ILE for reuse.</td>
<td>Immediate next steps: (1) take the clearest agent from your 4 prototypes, (2) write an output schema for it (describe in comments: "this agent must output { field1: type1, field2: type2, ... }"), (3) review the Agno Investment Team source code and find an agent with an explicit Pydantic output model—study the schema structure, (4) rewrite your agent's output using Pydantic (even if it's just a copy from a template), (5) in a new task, write the input AND output schemas BEFORE coding the agent, (6) reflect: did schema-driven development clarify your thinking about what the agent owns?</td>

<td>Success occurs when the User explicitly maps ILE chapters to agent roles and uses this mapping as the foundation for decomposition: (1) for a new task, identify which ILE chapters apply, (2) for each applicable chapter, name an agent responsible for that cognitive phase, (3) chain the agents in the order chapters appear in the ILE, (4) define handoff schemas (what does the previous phase output that the next phase consumes?), (5) build agents following this ILE-mapped architecture, (6) validate that each agent's scope aligns with its chapter's cognitive function.</td>

<td><strong>UBS.UD.UB: Prototypes are single-session throwaway — momentum doesn't compound</strong></td>
<td>This is the blocker preventing the User from leveraging the UBS.UD driver (Cursor + 4 prototypes). The User has capability assets but is not systematizing them. <td>The driver overcoming this blocker is the User's developing discipline of documentation and pattern extraction. This is not about Cursor or Python—it's about the User's willingness to invest time in translating implicit knowledge (intuition from a working prototype) into explicit knowledge (a reusable pattern documented in the ILE).
