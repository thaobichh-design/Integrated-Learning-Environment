| **Hierarchical Decomposition Thinking (Systems Thinking)** | **Relevance (Topic 2 · UDS):** This is the root driver of effective AI orchestration and the subject of Topic 2. It converts domain knowledge into agent architecture — which specialist roles to create, where boundaries sit, and what flows between them. For the Learner, mastering it means each new domain stops resetting understanding to zero; skill compounds across builds. Without it, agents are designed by gut feel and the OE vision stays out of reach. With it, any complex domain can be systematically translated into a working multi-agent system that generalises across domains. | The mental capacity to identify — for any complex domain process — (1) the distinct specialist roles it contains, (2) where clean boundaries between roles sit, (3) what structured outputs flow between them, and (4) what triggers each handoff. IS architectural thinking: domain-agnostic, transferable across domains, independent of domain knowledge. IS NOT domain expertise (knowing what to do). IS NOT mastered by thinking about decomposition — only by repeatedly decomposing across multiple build cycles until the role/boundary/handoff pattern becomes recognisable in any new domain. | Apply decomposition before every build — before writing any code, draw the role map: list each agent, its ONE job, its input, its output, and the handoff trigger to the next agent. The role map IS the decomposition act. Doing it consciously every build is how the skill accumulates. | Deliberate post-build reflection — after each build cycle, extract ONE explicit decomposition principle in writing: "what boundary decision made this work, and why?" Conscious extraction converts tacit building experience into a transferable mental model that is available for the next build. This is the driver of the driver. | Each build forces a boundary decision → post-build reflection extracts the principle → principle is written explicitly in the ILE Learning Book → next build applies that principle → cleaner boundaries → better outputs → higher confidence → richer reflection. Compounding loop: each cycle makes the next cheaper and more precise. No reflection step = no compounding. | P3 (enables UDS.UD): After every build, extract ONE decomposition principle in explicit language before advancing to the next phase. Conscious extraction is what converts doing into knowing — without it, experience accumulates but skill does not compound. | **Infrastructure:** Agno reference architectures (expert-built multi-agent examples to study where decomposition boundaries were placed and why). **Workspace:** ILE Learning Book (structured place to write extracted principles after each build). **Intelligence:** AgentOS dashboard (observe agent behaviour at runtime to validate whether boundary decisions held up in production). | Dedicated reflection time after each build — not just build-and-ship. A real domain problem with enough complexity to force genuine boundary decisions (a trivial task cannot teach decomposition). A sparring partner (ILE agent) to challenge extracted principles: "Is that the real reason the boundary worked?" | User builds without reflecting — ships working code but skips the extraction step. Experience accumulates, but each new domain still feels like starting from zero because no transferable principle was articulated. Decomposition skill stays tacit and never compounds. | Pattern blindness — the User cannot yet see the universal decomposition pattern (every complex system has: specialist roles → structured flows → clean handoffs) that holds across all domains. Without pattern recognition, every domain resets understanding to zero and compounding is impossible. Seed for Topic 2 deep dive. | Pattern blindness → each new domain feels unique and unfamiliar → User rebuilds decomposition understanding from scratch every time → effort stays linear, not compounding → multi-domain OE vision exhausts the User before generalisation is achieved → abandons the system before it compounds. | "Each domain is unique — I must learn its specifics before I can design agents for it." True at the surface level, lethal when it prevents the User from seeing the role/flow/handoff pattern that repeats underneath every domain's surface differences. | Domain-specific tutorials and documentation that teach WHAT to do in one domain without revealing the universal decomposition pattern beneath it. Consumption of these without ILE reflection locks in domain-specific thinking rather than pattern thinking. | Learning one domain in isolation with no cross-domain comparison. No structured reflection practice. No ILE framework applied to extract transferable principles. The longer the User studies only one domain in depth, the harder it becomes to see the repeating pattern beneath it. | Use ILE to decompose a domain already well understood (a past job, a familiar process) into agent roles — not to build, but to practise the decomposition act in safe familiar territory where boundary decisions feel obvious. Pattern recognition emerges from contrast: once the same role/flow/handoff structure is seen across two very different domains, the universal pattern clicks and stays. | Topic 2 Page 1 (UB Layer): deep dive on col 10 (UBS = Pattern blindness).
